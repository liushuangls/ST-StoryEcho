# StoryEcho · 剧情回响

StoryEcho 是一个面向 SillyTavern 长剧情聊天的上下文管理扩展。

它把长聊天整理成“分批阶段总结 + 近期原文 + 可选的按需剧情召回”。只配置 LLM 就能使用最小原文窗口和历史阶段总结；需要更精确的窗口外事实时，再开启“剧情记忆与召回”并配置 Embedding。

设置页只保留两个功能开关：

1. `启用 StoryEcho 上下文管理`：自动维护最小原文窗口和历史阶段总结，只依赖 LLM；
2. `启用剧情记忆与召回`：自动抽取、整理、向量同步、检索和请求级注入，额外依赖 Embedding。

关闭第二个开关不会删除已有剧情元数据或向量，也不影响阶段总结；它只会停止记忆层的后台维护和本轮使用。重新开启后会校验历史与向量配置并安全追赶。

## 核心目标

- 明确限制每次发送的原始聊天历史长度；
- 使用有覆盖游标的分批阶段总结维持长期剧情脉络；
- 将历史对话抽取为结构化剧情记忆，而不是复制原文；
- 使用有界、只读的角色卡与批次命中世界书参考帮助抽取器消歧；
- 使用 SillyTavern Vector Storage 在服务端保存向量和执行语义检索；
- 默认复用 SillyTavern 主连接和当前向量来源；
- 可分别配置自定义 OpenAI 兼容 LLM、OpenAI 兼容 Embedding 与火山方舟多模态 Embedding；
- 与 MVU、世界书和其他状态系统解耦；
- 让上下文管理与可选的剧情记忆层可统计、可调试、可独立启停，并查看最近一次真实角色请求的 Token 构成。

## 请求与存储方案

StoryEcho 采用无需额外服务端插件的组合方案：

```text
自定义 LLM
浏览器扩展 -> SillyTavern自带Chat Completions后端 -> 外部LLM

自定义 Embedding
浏览器扩展 -> SillyTavern内置/proxy -> 外部Embedding接口
             -> 预生成向量 -> SillyTavern Vector Storage
                                （服务端保存和检索）
```

- 自定义 LLM 使用酒馆自带的 `/api/backends/chat-completions/generate` 转发，外部请求由 SillyTavern 服务端发出；
- 自定义 Embedding 自动把外部地址路由到 SillyTavern 内置 `/proxy/`，由酒馆服务端请求，因此不要求外部接口允许浏览器 CORS；
- 浏览器只负责生成自定义向量，不在前端维护向量数据库或执行余弦检索；
- 向量仍交给 SillyTavern Vector Storage，因而可随酒馆数据在多端共用。

## 项目状态

项目处于 Alpha 测试阶段。目前已有两种 LLM Provider、OpenAI 兼容与火山方舟多模态两种外部 Embedding、统一前台/手动/后台优先级队列、按配置满批抽取、回复后单批后台整理、3000 Token 受控抽取参考、六类分类抽取、完整轮次自适应拆批、可见剧情正文清洗、稳定身份资料抽取、字段级来源落地、Assistant推断降级、候选质量门槛、六类事件整理动作、User事实优先、同状态槽多目标失效、承诺稳定逻辑键、分级阶段总结、事实核验召回、LLM 查询改写与规则降级、可查看/编辑/删除的聊天元数据、Vector Storage 增量同步、滑动窗口、生成拦截、跨阶段当前状态校正、最近一次请求 Token 构成、运行统计和调试报告。Token 卡片读取 SillyTavern 当前聊天本地保存的最新提示词明细，单独列出最近原文、阶段总结、当前状态校正和动态召回，再展示角色卡、世界书、系统预设、示例与其他扩展占比；酒馆未保存分类计数时会明确降级为可识别文本估算。合并记忆会按完整 `sourceHistory` 判断是否越过窗口边界，避免“最新确认仍在窗口内”时漏召回更早的关键状态。

主连接用于后台总结、抽取、整理和查询改写时会读取酒馆当前 API 来源与精确模型名，并临时采用轻量推理设置，不改变正常角色聊天的推理预设。`deepseek-*` 会优先使用 JSON Object，其他模型优先 JSON Schema，失败后按真实能力降级。召回注入使用结构化事件、当前结果、状态变化、实体和知情范围重新渲染，不直接依赖模型生成的第一人称摘要。

正常角色请求中的顺序固定为：

```text
角色卡、世界书、系统提示
历史阶段总结
跨阶段变更后的当前状态校正
尚未满一批的窗口外原文
最小保留窗口内的近期原文
本轮动态召回（请求级 system，不写入聊天记录）
当前用户原始输入
```

总开关开启后，阶段总结始终自动维护，不再另设“启用总结”或“自动总结”开关。最小保留窗口 `W` 决定最近至少保留多少原文；总结间隔 `N` 决定窗口外每累计多少完整轮次生成一条新总结；总结窗口 `S` 决定一次角色请求最多携带最近多少条阶段总结。自动流程不会在后续批次中反复重写已有总结，用户仍可在设置页查看、编辑或删除单条总结；若 User 明确结束上一剧情阶段并开始新阶段，边界前内容会提前封成独立总结，避免一条总结混入两个阶段。新总结固定分为“已确认剧情、当前状态、未解决线索、角色主张与推测、已失效或否定事实”，避免推断混进长期事实；标题不完整时不会推进总结游标。

阶段总结保存在当前聊天的 `chatMetadata.story_echo.stageSummary.entries` 中，并随聊天 JSONL 保存。删除操作不会修改任何聊天原文：删除最新总结会回退覆盖位置，使该批原文重新参与后续请求；删除更老的总结只停用该条并保留覆盖范围，不会把很老的原文重新送回模型，也不会影响后续总结。

剧情抽取同样按“每批抽取轮数”累计完整的“用户 + AI”轮次，默认 5 轮才调用一次；不足一批时不调用抽取 LLM，也不裁剪这段原文。AI 回复完成后后台最多整理一个抽取批次和一个总结批次；下一轮生成前不会同步抽取、总结或写向量，后台未追上的原文会继续保留，因此不会为了追赶索引阻塞角色回复。密集批次先在本地展开并质检，再从合格项中按状态变化、承诺、揭示等长期价值排序，单批最多保留 20 条，并为已有的不同记忆类型各保留至少一个位置，避免状态事实挤掉关键因果剧情；较小批次数不会再被误当成更小候选上限。动态召回默认最多 3 条，并会先过滤分数过低或与最佳结果差距明显的尾部候选；显式要求分别核对或分项回答多个实体时可在 Token 预算内临时扩展，最多 8 条，并可从排序阈值外补回被明确点名的实体。用户明确声明上一剧情阶段结束并开始新的独立阶段后，普通续写默认隔离较早阶段的自动记忆与总结；置顶、人工编辑、明确点名的旧实体或直接询问较早阶段仍可跨阶段使用。边界识别兼容篇章、任务、旅程、冒险、故事线等不同题材，并且不会把普通换场、时间推进或新线索误当成阶段切换。

抽取模型收到 StoryEcho 自己的抽取 system prompt、该批真实聊天楼层，以及一个总预算默认 3000 Token 的受控只读参考块。Assistant 楼层会先清除隐藏思考、HTML注释和常见展示外壳，仅发送可见剧情正文；User楼层保持原文。角色部分仅取身份、Persona、描述、性格与场景；世界书部分只从当前聊天实际关联的书中选择被这批旧消息直接命中的最多 5 条。角色卡 system/jailbreak、作者注释、示例对话、欢迎语、主提示词和 Prompt Manager 预设始终不传。参考块只能用于实体和别名消歧，每条记忆必须引用真实聊天中的 `sourceMessageIds`；具体人物专名和编号还要能在引用正文、Assistant角色名或此前已落地实体中找到依据，不能给匿名人物凭空补姓名。Assistant推断若被误标为confirmed会在本地降为inferred。主连接后台任务在酒馆提供相应字段时会临时关闭重推理并采用确定性采样；内部LLM空响应会扩大有界输出预算重试一次。结构化响应会先尝试无损本地JSON语法修复，仍失败才付费降级；若此时已有前台生成排队，后台先让行并在真实回复后重试。文本补全 API 仍可能沿用 instruct 格式。其他扩展若监听酒馆的 raw prompt 事件并主动改写请求，则仍可能影响后台调用。

冲突时使用“当前用户输入 > 近期原文 > 动态召回 > 当前状态校正 > 阶段总结”的新旧优先级。当前用户消息本身不会被修改；近期User明确更新会遮蔽窗口外同槽旧记忆，单纯提问不会触发遮蔽。Assistant无时间推进依据的裸冲突不会覆盖User已确认事实，但后续楼层中明确发生的已确认状态转移会作为正式剧情推进。阶段总结还会收到同批、带当前版本真实来源楼层的有界校正账本；只采用 active/resolved 事实，空间不足时优先保留较新的明确转移，避免已被User否定的Assistant推测或已取代旧值继续进入长期总结。

当用户明确要求“只回答当前事实/不要推测/没有已确认记录”，或询问具体当前位置、持有者、知情者等封闭事实时，StoryEcho进入事实核验模式：只让confirmed记忆参与检索，并从阶段总结中省略未解、推测和失效段。普通剧情续写和开放式调查仍可召回其他事实等级。

消息编辑、删楼、Swipe 与分支切换已经采用保守失效重建。设置页的阶段总结与“剧情记忆元数据”都可搜索、分页查看、修改和单条删除；人工修改会被保护，不再由自动整理覆盖。向量写入、删除、重建或待同步队列变化后，状态栏会自动刷新服务端向量数；普通生成不会反复查询计数。抽取规则升级或旧批次曾返回空结果时，可用“重建自动元数据”重新处理窗口外历史并保留人工修改项。

## 安装

只需在 SillyTavern“扩展程序 → 安装扩展程序”中安装：

```text
https://github.com/liushuangls/ST-StoryEcho
```

不需要安装或启用 SillyTavern Server Plugin。

使用自定义 Embedding 前，需要在 SillyTavern 的 `config.yaml` 中启用内置代理并重启：

```yaml
enableCorsProxy: true
```

StoryEcho 会在请求时自动添加 `/proxy/`；设置中仍填写正常的外部 Base URL，不要手动添加代理前缀。默认继承 Vector Storage 来源时不需要开启此选项。

## 自定义接口与 Key

LLM 与 Embedding 的 Base URL、模型和 API Key 都保存在当前 SillyTavern 用户的 `extensionSettings` 中，并由酒馆的设置同步机制持久化。刷新页面后会自动恢复；使用同一酒馆用户数据的其他客户端也能读取。

这带来的安全边界必须明确：Key 是明文设置，不是 SecretManager 密钥。它对同页面的其他 UI 扩展和能读取用户设置的人可见。建议使用限额、限权、可撤销的独立 Key。调试报告、聊天元数据和 Vector Storage 不记录 Key；开启调试后，报告会包含有界的抽取参考预览，用于检查世界书或角色设定是否污染记忆。

### 火山方舟 Embedding

在 StoryEcho 设置中选择“火山方舟多模态Embedding”，填写：

```text
Base URL: https://ark.cn-beijing.volces.com/api/v3
模型: doubao-embedding-vision-251215 或多模态推理接入点 ep-m-…
API Key: 方舟API Key
```

StoryEcho 会调用 `/api/v3/embeddings/multimodal`，把每段剧情文本作为独立的 `{ type: "text" }` 输入，最多并发 4 个请求，并从 `data.embedding` 读取向量。点击“测试火山Embedding连接”验证酒馆代理与方舟配置；测试成功后，生成的向量仍写入当前聊天的 Vector Storage 集合。

方舟 Coding Plan 的 `https://ark.cn-beijing.volces.com/api/coding/v3` 属于 OpenAI 兼容协议，应继续选择“自定义OpenAI兼容接口”，并使用套餐提供的模型名。

## 开发与安装产物

```bash
npm install
npm run check
```

`manifest.json` 加载 `dist/index.js`，并为 JS/CSS 入口附加与扩展版本一致的查询参数，避免更新后继续命中旧浏览器缓存。SillyTavern 通过 Git 地址安装扩展时不会执行前端构建，因此仓库会提交由源码生成的 `dist/`；修改源码后必须重新构建并一起提交产物，发布新版本时也必须同步更新入口查询参数。

## 设计边界

StoryEcho 不负责：

- MVU 变量或角色状态管理；
- 世界书内容管理；
- 剧情导演、自动续写或修改实际发送的用户输入；
- 保存整份原始聊天的副本；
- 把旧消息重新伪装成当前的 user/assistant 对话。
