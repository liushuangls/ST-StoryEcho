# StoryEcho 产品需求文档

## 1. 产品定义

StoryEcho 是一个“滑动窗口上下文 + 历史剧情召回”扩展。

每次正常聊天生成时，它只让最近 `N` 轮原始聊天进入提示词，同时根据当前用户输入，从窗口之外的历史中检索已经抽取的关键剧情记忆，并将选中的记忆插入最近聊天历史之前。

StoryEcho 的重点不是保存原文或普通摘要，而是把聊天转换成结构化、可维护、方便检索的剧情事件。

## 2. 用户问题

长聊天通常会遇到以下问题：

1. 原始聊天历史持续增长，消耗大量 Token。
2. 依赖模型上下文上限自动截断时，用户无法确定哪些消息实际被发送。
3. 普通总结容易丢失人物知情范围、承诺、物品归属和因果关系。
4. 直接对原文做向量检索，会召回大量描写相似但剧情价值低的片段。
5. 召回旧 user/assistant 消息容易使模型混淆当前时间线。

## 3. 目标

### 3.1 必须实现

- 可配置只发送最近 `N` 轮或最近 `N` 条消息。
- 当前用户输入始终保留，不计入 `N`。
- 只从本次未作为原始消息发送的历史范围中召回。
- 将旧历史抽取为原子化剧情事件。
- 支持事件新增、合并、更新、解决、取代和忽略。
- 对剧情事件生成独立的检索文本与注入文本。
- 使用 SillyTavern Vector Storage 保存向量并在服务端检索。
- 默认使用 SillyTavern 主连接执行剧情抽取和整理。
- 可切换到自定义 OpenAI 兼容 LLM，配置 Base URL、模型和 API Key。
- 提供本次裁剪与召回检查器。
- 处理消息编辑、删除、Swipe 和聊天分支。
- 检索或抽取失败时放行正常生成，不阻塞聊天。

### 3.2 不在范围内

- 不替代 MVU 或其他变量框架。
- 不维护完整人物属性数据库。
- 不自动改写世界书。
- 不接管剧情规划或扮演逻辑。
- 不永久保存原始聊天副本。
- 不保证抽取结果绝对正确；用户必须能查看、修改或删除记忆。

## 4. 术语

### 4.1 楼

默认一楼表示一个完整对话轮次：

```text
用户消息 + 随后的 AI 回复
```

高级设置允许改为按消息条数计算，以适应群聊、连续生成和多段回复。

### 4.2 最近窗口

每次生成时保留的最近 `N` 楼原始消息。角色卡、系统提示、世界书和其他扩展提示不属于最近窗口，不应被 StoryEcho 当成聊天消息删除。

### 4.3 剧情记忆

从一个历史场景中提取出来的原子化事件。它不是原文副本，也不是整段流水账总结。

### 4.4 检索文本

专门用于关键词和向量搜索的冗余文本，可以包含实体别名、同义表达、原因、结果和未解决问题。不会直接发送给角色模型。

### 4.5 注入文本

发送给角色模型的简洁自然语言，只表达当前生成所需的已确认剧情背景。

## 5. 用户流程

### 5.1 正常生成

```text
用户发送消息
  -> 确认剧情记忆索引已覆盖到最近窗口边界
  -> 构造检索查询
  -> 从当前聊天的 Vector Storage 集合检索候选事件
  -> 应用实体、状态、知情范围和重要度过滤
  -> 在 Token 预算内选择事件
  -> 删除窗口之外的原始聊天消息
  -> 在最近窗口之前插入“相关往事”系统块
  -> 继续正常生成
```

### 5.2 后台抽取

AI 回复完成后，新消息进入待处理缓冲区。达到以下任一条件时创建剧情块：

- 累计 2～4 个对话轮次；
- 地点、时间、主要参与者或话题明显变化；
- 最近窗口边界即将越过尚未处理的消息。

后台抽取不应阻塞当前回复展示。若下一次生成时索引尚未覆盖裁剪边界，扩展最多同步补做一个小分块；仍未覆盖时暂时保留完整聊天。用户可通过“处理窗口外历史”主动完成积压，任何情况下都不能静默丢失历史。

## 6. 剧情事件抽取要求

### 6.1 应当保留

- 已发生且影响后续理解的事件；
- 人物、物品、地点、阵营或关系的状态变化；
- 承诺、任务、目标、命令、约定、期限和代价；
- 身份、秘密、动机和过去的揭示；
- 线索、伏笔和未解决问题；
- 难以逆转的结果；
- 后续再次被提及、证明具有持续价值的信息。

### 6.2 应当忽略

- 普通寒暄；
- 没有后果的动作和环境描写；
- 重复表达相同情绪或事实；
- 未被确认的随意推测；
- 只影响当前一句回复的修辞细节；
- 与已有记忆完全重复且没有新信息的内容。

### 6.3 事实约束

- 不得补全原文没有的信息。
- 必须区分客观事实、角色声称、模型推断和不确定信息。
- 必须记录哪些角色知道该信息。
- 每条事件只表达一个主要变化或关系。
- 每条事件必须保留来源消息范围和来源哈希。

## 7. 事件生命周期

新候选事件与现有相似事件比较后，只能执行以下操作：

- `CREATE`：创建新事件；
- `MERGE`：向现有事件补充信息；
- `UPDATE`：更新仍在持续的事件；
- `RESOLVE`：标记任务、伏笔或冲突已解决；
- `SUPERSEDE`：新事实取代旧状态；
- `IGNORE`：丢弃无价值或重复信息。

事件状态包括：

- `active`：仍可能影响后续剧情；
- `resolved`：已经解决，可在明确追溯过去时召回；
- `superseded`：已被新事实取代，默认不注入；
- `invalid`：来源消息已删除、编辑或不属于当前 Swipe。

## 8. 检索策略

检索查询由以下内容组成：

- 当前用户输入；
- 最近 1～2 轮场景；
- 当前提到的人物、地点、物品和目标；
- 可选的 MVU 当前场景适配器结果，但核心不得依赖 MVU。

候选排序综合：

- Vector Storage 返回顺序；
- 实体和别名精确匹配；
- 未解决伏笔匹配；
- 事件重要度；
- 当前场景连续性；
- 事件状态惩罚；
- 最近已经召回但已在近期回复中体现的降权。

召回事件按剧情原始顺序输出，不按相似度顺序输出。必要时沿因果关系补充前置事件。

## 9. 注入规则

默认注入结构：

```text
<story_echo>
以下是与当前场景相关的较早事件，不代表当前正在发生：
- 较早时，用户把银色钥匙交给了林雨，目前仍由她保管。
- 林雨答应午夜前不使用钥匙；这项承诺尚未得到验证。
</story_echo>
```

位置：角色卡、系统提示和世界书之后，最近原始聊天窗口之前。

不得把旧剧情重新构造成多条 user/assistant 消息。默认使用中立系统提示；无法支持中间 system 消息的 API 由 SillyTavern 完成格式适配。

## 10. 设置需求

### 10.1 基础设置

- 启用 StoryEcho；
- 最近窗口大小；
- 计数单位：轮次或消息；
- 历史召回最大 Token；
- 最多召回事件数；
- 语义相关性阈值；
- 后台自动抽取开关。

### 10.2 LLM设置

默认选项：`SillyTavern 主连接`。

自定义选项：`OpenAI 兼容接口`，包含：

- Base URL；
- API Key；
- 模型名；
- 测试连接按钮。

安全要求：

- Base URL、模型名和非敏感参数可以保存在 `extensionSettings`。
- API Key 不得明文写入 `extensionSettings`、聊天元数据或角色卡。
- Base URL不得携带用户名、密码或查询参数，避免凭据混入持久化设置。
- 纯 UI 扩展模式下，Key 默认只保存在当前页面内存，刷新后需要重新输入。
- 后续可选服务端插件负责安全保存 Key 和代理自定义请求。
- UI 必须明确显示 Key 当前是否已加载，而不能回显完整 Key。

### 10.3 Vector Storage设置

- 默认复用 SillyTavern Vector Storage 当前来源与模型；
- 显示当前聊天集合 ID；
- 显示事件数、已向量化数量和待处理数量；
- 提供重建当前聊天索引；
- 提供清除当前聊天向量；
- 更换 Embedding 来源或模型后自动重建，不混用旧向量。

## 11. 检查器

每次生成至少记录：

- 保留的最近消息范围；
- 被裁掉的原始消息范围；
- 检索查询摘要；
- Vector Storage 返回的候选；
- 最终选择和排除的事件及原因；
- 注入文本；
- 各部分估算 Token；
- 失败和降级信息。

用户可以：

- 固定一条事件；
- 禁止一条事件自动召回；
- 修改注入文本；
- 查看来源消息；
- 删除并重新抽取事件。

## 12. 失败与降级

- 抽取失败：保留待处理区，稍后重试。
- Vector Storage失败：不召回，保留最近窗口并继续生成。
- 最近窗口边界之前存在未抽取历史：暂时扩大窗口，避免信息静默丢失。
- 自定义 LLM失败：显示错误；若用户开启回退，则改用主连接。
- 记忆 JSON 校验失败：拒绝写入，保留原数据。
- 提示词拦截失败：放行原始生成，不破坏聊天数据。

## 13. MVP验收标准

1. 安装后可在扩展面板看到 StoryEcho 设置。
2. 默认 LLM Provider 是主连接。
3. 可选择自定义 Base URL、模型和会话内 API Key。
4. 每个聊天具有独立记忆集合。
5. 可手动对指定消息范围执行剧情事件抽取。
6. 抽取结果通过 Schema 校验后写入聊天元数据。
7. `retrievalText` 可写入独立 Vector Storage 集合并被查询。
8. 正常生成只保留配置的最近窗口。
9. 旧剧情以单个中立记忆块插入最近窗口之前。
10. 检查器能展示本次裁剪和召回结果。
11. 编辑、删除和 Swipe 会使相关记忆失效或重建。
12. 任一扩展步骤失败都不会阻止正常聊天。
