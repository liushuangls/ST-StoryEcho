# StoryEcho 调度与结构化记忆管线改造方案

状态：已批准，按本文顺序实施。

本文记录 2026-07-19 在“陌白·福尔摩斯”真实聊天中发现的并发、结构化输出、过度原子化和状态整理问题，以及对应的实现方案。角色正常回复的最大输出配置不属于本次改造，继续由 SillyTavern 控制；测试时保持 8K 上限。

## 1. 改造目标

1. 后台任务运行时到来的真实用户生成必须进入等待队列，不能绕过 StoryEcho。
2. 所有需要 JSON 的内部任务按模型在 JSON Object 与 JSON Schema 之间选择首选模式，最后才用普通模式；Provider 报错、空输出、解析失败或本地校验失败时自动降级。
3. 抽取结构按记忆类型分类。剧情和冲突保持完整因果片段，不再被通用分句器原子化；只有可独立更新的状态、关系和承诺按稳定逻辑槽拆分。
4. 位置、持有者、知情范围、否定事实和承诺完成优先使用确定性整理，LLM 只处理语义含糊的剧情重复关系。
5. 任何后台失败都不得推进覆盖游标或裁掉未覆盖原文。

## 2. 单一任务协调器

新增一个运行时 `StoryEchoTaskCoordinator`，统一协调三类工作：

- `foreground`：正常发送、Regenerate 和 Swipe 的请求级上下文准备；
- `manual`：用户主动处理历史或重建自动元数据；
- `background`：AI 回复后的自动抽取、阶段总结和后续同步。

### 2.1 调度规则

1. 已开始的任务不抢占。
2. 后台任务运行时到来的前台任务进入队列。
3. 当前任务结束后，前台任务排在所有尚未开始的后台任务之前。
4. 后台任务按聊天和目标楼层合并，只保留最大待处理目标，避免重复排队。
5. 前台上下文准备开始时取得“前台租约”。拦截器返回后租约继续有效，直到真实 AI 回复完成、取消、失败或超时；租约期间不得启动新后台任务。
6. 等待任务执行前重新核对聊天 ID、历史修订号和已索引前缀哈希。切换聊天、编辑、删楼、Swipe 或切分支后必须重新协调，不能使用排队前的旧状态。
7. 设置页显示当前任务、等待中的前台/后台数量、前台等待时间和租约状态。

### 2.2 内部生成识别

现有模块级 `internalGenerationDepth` 只能表示“此刻存在内部调用”，无法判断当前拦截事件是否由该内部调用产生。它会让并发到来的真实用户生成直接跳过插件。

改造后，每个内部 `generateRaw` 请求携带唯一、不可由普通聊天自然产生的 StoryEcho 请求标记。生成拦截器只跳过包含有效标记的内部请求；普通用户请求即使在内部请求尚未结束时到来，也必须进入协调器。全局内部深度不再作为跳过真实生成的依据。

### 2.3 安全行为

- 前台最多等待当前一个已开始任务，不等待整个后台积压队列。
- 当前后台批次失败时，前台仍从最新已提交状态构造请求；未覆盖历史继续作为原文发送。
- 前台租约异常未释放时使用有界看门狗恢复，但看门狗不得推进索引或总结游标。

## 3. 兼容性优先的结构化调用

新增统一接口：

```ts
interface StructuredCompletionRequest<T> {
  request: LlmRequest;
  schema: Record<string, unknown>;
  parse(raw: string): T;
}

completeStructured<T>(input: StructuredCompletionRequest<T>): Promise<T>;
```

调用和解析必须在同一层完成，因为“Provider 请求成功但 JSON 无法解析”同样需要触发普通模式回退。

### 3.1 Provider 内重试顺序

每个 Provider 按模型和实际能力执行：

```text
deepseek-*：JSON Object -> JSON Schema -> 普通模式
其他模型：JSON Schema -> JSON Object -> 普通模式
主连接：从酒馆上下文读取当前 `mainApi`、来源与精确模型名，再使用同一模型优先级；实际来源不支持某一层时跳过该层

JSON Object 和普通模式都携带明文 Schema + 完整 JSON 示例
自定义连接三种模式都失败且开启 Provider 回退：在主连接按其实际能力重复
```

“失败”包括：

- HTTP、超时或 Provider 异常；
- 空输出（DeepSeek 官方说明 JSON Output 偶发会返回空 `content`）；
- JSON 截断或无法解析；
- 根结构、必填字段或枚举不符合 Schema；
- 本地解析后没有得到合法项目。

JSON Object 请求必须发送 `response_format: { "type": "json_object" }`，并在 system 或 user Prompt 中包含 `json` 字样、完整 Schema 与一份合法 JSON 输出示例。自定义 OpenAI 兼容连接通过 SillyTavern 的 `custom_include_body` 将该参数合并进最终模型请求。主连接从 `getChatCompletionModel()` 取得模型；原生 DeepSeek 来源利用酒馆把 `json_schema` 请求转换为 JSON Object 的后端能力，自定义 DeepSeek 主来源则只在本次内部请求的 `CHAT_COMPLETION_SETTINGS_READY` 阶段临时追加 `custom_include_body`。无法真实发送 JSON Object 的来源必须跳过该层，不能把纯提示词模式误报为 JSON Object。

模型名按最后一个路径段、忽略大小写匹配，因此 `provider/deepseek-v4-pro` 也使用 DeepSeek 顺序。未知模型默认优先约束更强的 JSON Schema。

旧 `strictJsonSchema` 开关不再阻止统一调用层执行三阶段协商。结构化输出能否接受由本地 Parser/Validator 决定，而不是仅以 HTTP 200 为准。

### 3.2 二次失败与拆批

同一 Provider 的 JSON Object、Schema 与普通模式均失败后：

- 抽取任务把完整轮次批次按轮次边界二分；
- 不拆开一组 User + Assistant；
- 单轮仍失败时保留原文，并以有界退避等待手动或后续重试；
- 同一个失败分块不能在每条用户消息到来时立即重复请求；
- 失败输出只在调试轨迹中保存有界预览，不保存 Key。

新增指标：JSON Object 请求/失败、Schema 请求/失败、普通模式重试/成功、Provider 回退、自适应拆批和冷却跳过次数。

## 4. 分类抽取结构

抽取模型返回分类根对象，而不是一组共享 15 个文本字段的通用对象：

```json
{
  "episodes": [],
  "stateFacts": [],
  "relationships": [],
  "commitments": [],
  "revelations": [],
  "clues": []
}
```

解析后仍归一化为现有 `StoryMemory`，因此不立即改变聊天元数据 Schema 或 Vector Storage 集合版本。

### 4.1 粒度规则

| 分类 | 映射类型 | 粒度与生命周期 |
| --- | --- | --- |
| 剧情片段 | `event` / `conflict` | 同一场景、目标和因果链保持一条；不按参与者、物品或标点拆分 |
| 状态事实 | `state_change` | 每个“完整实体 + 规范状态槽”一条；同槽新值确定性取代旧值 |
| 关系变化 | `relationship_change` | 每对实体的一种关系一条；按关系槽更新 |
| 承诺任务 | `commitment` | 每个承诺或任务一条；使用稳定逻辑键推进未完成、完成、取消或失败 |
| 揭示秘密 | `revelation` | 一个完整命题一条；知情范围和真伪可附带状态槽，不按句号机械拆分 |
| 线索证物 | `clue` | 同一证物及其直接含义保持一条；不同证物才拆分 |

同一段原文可以同时产生一条完整剧情和若干可变状态。例如“众人在煤窖取得罗盘和胶片，陌白收入风衣暗袋，旧砖洞已空”保留一条完整取得证物的剧情，同时生成罗盘位置、胶片位置和砖洞内容三个独立状态槽。剧情本体不被拆碎。

### 4.2 确定性文本生成

模型不再重复输出高度相似的 `event`、`retrievalText` 和 `injectionText`。分类结果只提供必要事实字段；本地按类型生成检索和注入文本：

- 剧情：时间、地点、参与者、行动、原因和结果；
- 状态：实体、属性、旧值和当前值；
- 关系：双方、关系类型和变化；
- 承诺：承诺者、对象、行动和生命周期；
- 线索：证物、观察和被支持的含义；
- 揭示：完整命题、真伪与知情范围。

所有项目继续要求 `sourceMessageIds`，参考上下文不能成为证据。

### 4.3 数量与降级

候选总数使用随批次变化的上限：

```text
min(12, max(6, 完整轮次数 * 3))
```

通用 `atomicizeMemoryCandidates` 替换为 `normalizeCandidatesByType`：

- `event/conflict/clue/revelation` 保持模型给出的完整项目；
- `state_change` 只按规范状态槽拆分；
- `relationship_change` 只按关系边拆分；
- `commitment` 只按稳定承诺键拆分。

## 5. 确定性整理优先

LLM 整理之前先执行本地规则：

- 同一状态槽、同值：`MERGE` 或 `IGNORE`；
- 同一状态槽、新值：`SUPERSEDE`；
- 同一关系槽：`UPDATE` 或 `SUPERSEDE`；
- 同一承诺进入完成态：`RESOLVE`；
- 明确否定或纠正：失效被纠正事实并保留新权威事实；
- 同一来源哈希和事实文本：`IGNORE`。

一次状态更新必须取代该逻辑槽的全部活跃重复项，而不只取代模型选中的一个目标。人工编辑和固定记忆继续受到保护。

只有以下歧义进入 LLM：

- 两个剧情片段是否为同一事件；
- 两条线索是重复、补充还是独立证物；
- 没有稳定状态槽的自然语言修正；
- 名称或别名是否指向同一实体。

整理 Prompt 只发送紧凑字段，输出只包含候选索引、动作和目标 ID。整理也使用 JSON Object、Schema、普通模式三级回退。整理最终失败时只允许模糊的剧情类候选保守 `CREATE`；状态、关系和承诺的当前值不能依赖整理 LLM 才保持一致。

## 6. 阶段总结与身份

- 阶段总结负责时间线、目标、关系进展和阶段局势，不承担“当前持有者/当前位置”的唯一权威职责；当前状态由有效状态记忆在总结之后校正。
- 总结输入明确标注 UI Persona、剧情用户角色和 AI 角色，不能把 SillyTavern Persona 名直接当成剧情角色名。
- 无可靠映射时使用“用户角色”，不得新增姓名、种族、性别、身份或亲属关系。
- 总结提交前检查新增专名和稳定身份属性是否能在原文或只读角色参考中找到依据；没有依据的内容删除或拒绝该总结。

## 7. 实施顺序

1. 任务协调器、前台等待和租约；
2. JSON Object、Schema 与普通模式三级回退；
3. 分类 Schema、Parser、Prompt 和类型归一化；
4. 确定性状态、关系和承诺整理；
5. 阶段总结身份约束和可观测性；
6. 自动化测试、构建和真实页面回归。

每一阶段完成后先运行对应测试；最终运行 `npm run check`。

## 8. 验收标准

- 后台运行时发送真实用户消息，前台进入队列且最终经过 StoryEcho；
- 同一时刻不出现 StoryEcho 后台主连接调用与正常角色生成重叠；
- JSON Object 失败时尝试 Schema，Schema 失败时再尝试同一 Provider 普通模式；
- 三种模式都失败时不推进游标、不裁掉原文，并按完整轮次拆批；
- 多人物、多物品的一段剧情保持为一条完整剧情记忆；
- 罗盘搬移后只有一个有效位置槽；
- “女仆发现尸体”被明确纠正后不再参与召回；
- 露西获救后救援承诺为 `resolved`；
- 两位艾达、多个 E.B.、白鸦庄园与白鸦酒馆不会串线；
- 角色正常回复最大输出设置仍为 8K；
- 现有人工编辑、分支隔离、删楼重建和 Vector Storage 同步测试继续通过。
